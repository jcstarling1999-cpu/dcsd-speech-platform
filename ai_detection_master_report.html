Chapter 1: What AI Writing Assistants, Detectors, and Humanizers Actually Are

The core taxonomy, the blurred boundaries, and why the arms-race metaphor is more than rhetoric.
Foundational

Professor View

AI writing assistants optimize production. They help users draft, revise, translate, simplify, paraphrase, cite, or reshape language. AI detectors, by contrast, are post-hoc classifiers or statistical tests that attempt to estimate whether text resembles machine-generated output. AI humanizers occupy the adversarial middle: they accept machine-generated or machine-assisted text, then rewrite it so that downstream detectors assign a more human-like score. The categories overlap because modern writing platforms increasingly bundle all three. Grammarly now offers core writing assistance, AI detection, and humanization; QuillBot combines paraphrasing, detection, and citation tooling. This makes product classification less about pure function than about the commercial workflow a vendor emphasizes.

[CITED] Historically, this ecosystem expanded rapidly after the mainstream arrival of ChatGPT in late 2022. A feedback loop followed: more generation led to more detection demand; more detection led to more humanizer demand; more humanization led detectors to claim bypass resistance; and writing suites responded by turning “responsible AI use” into a product category of their own.

Coder View

writer_assistant(input_text, context):
    return generate_or_revise(input_text, context)

detector(input_text):
    features = extract_features(
        perplexity=input_text,
        stylometry=input_text,
        classifier_embeddings=input_text
    )
    return probability_human_vs_ai(features)

humanizer(input_text, target="more_human"):
    rewritten = rewrite_for_variation(input_text)
    return optimize_against_detector_signals(rewritten)

Architecturally, these are different interfaces onto the same latent object: text distributions. Writing assistants sample from a model. Detectors estimate which distribution a sample came from. Humanizers attempt distributional migration, pushing text away from regions detectors associate with AI.

Plain English

A writing assistant helps make words. A detector guesses whether words look machine-made. A humanizer tries to rewrite machine-made words so they stop looking machine-made. They are different tools, but they are all circling the same basic question: what patterns make text feel predictably AI-like versus unpredictably human?

What We Can Only Infer

[INFERRED] The market is converging on workflow control, not just model quality. Vendors that can cover drafting, scanning, and rewriting inside one suite gain a data and retention advantage: users do not need to move text between separate tools. That helps explain why Grammarly, QuillBot, and others have expanded laterally into neighboring categories.

Chapter 2: How Language Models Generate Text

Transformers, token-by-token prediction, and why sampling settings change detectability.
Mechanics

Professor View

Modern large language models are autoregressive next-token predictors. Given a context sequence x₁ … xₙ, the model estimates a conditional probability distribution over the next token P(xₙ₊₁ | x₁ … xₙ). Transformer architectures implement this through stacked attention layers that allow each position to condition on prior positions. Generation is therefore not word-level planning in a human sense; it is repeated local probability estimation under a global context representation.

Sampling strategy matters. Greedy decoding always picks the single highest-probability token and thus tends to create highly regular, low-entropy text. Temperature rescales logits before softmax, flattening or sharpening the distribution. Top-k limits candidates to the k most likely tokens; top-p keeps the smallest set whose cumulative mass exceeds p. Beam search keeps multiple candidate continuations, often improving coherence but also reinforcing high-probability continuations. These choices directly affect downstream detector features because they alter variance, surprisal, repetition, and sentence rhythm.

Coder View

logits = model(context_tokens)

def sample_next(logits, strategy="top_p", temperature=0.8, k=40, p=0.92):
    logits = logits / temperature
    probs = softmax(logits)
    if strategy == "greedy":
        return argmax(probs)
    if strategy == "top_k":
        return sample(normalize(keep_top_k(probs, k)))
    if strategy == "top_p":
        return sample(normalize(keep_until_cumulative_mass(probs, p)))
    if strategy == "beam":
        return expand_multiple_best_paths(probs)

while not stop_condition:
    next_token = sample_next(model(context))
    context.append(next_token)

Lower-entropy decoding often yields text with stronger local smoothness and lower surprisal. That is one reason raw LLM outputs can look “too even” to detectors. Humanizers partly work by reintroducing variation that greedy or conservative sampling suppresses.

Plain English

The model is always making a guess about the next chunk of text. Then it adds that chunk, looks at the whole sentence again, and guesses the next one. If you make it conservative, it chooses safe, obvious next words. If you make it looser, it takes more risks. That difference changes whether the writing feels polished but robotic, or varied and human-like.

What We Can Only Infer

[INFERRED] Commercial humanizers probably do not rely on one trick. They likely combine prompt steering, paraphrase-specific fine-tuning, and post-processing rules that deliberately increase sentence-level variance, especially after the first pass of model rewriting. That would be the most economical way to combat detector features without training a wholly new generation stack from scratch.

Chapter 3: Tokenization — How Machines See Text

Why a token is not just a word, why BPE matters, and why token counts affect both context and cost.
Core Concept

Professor View

Tokenization is the mapping from raw text into the discrete vocabulary units a model can process. Character-level tokenization is flexible but inefficient; word-level tokenization fails on rare forms and morphology. Subword tokenization, popularized through methods such as byte pair encoding (BPE), WordPiece, and SentencePiece, balances compression and compositionality. Sennrich, Haddow, and Birch (2016) helped popularize BPE for subword modeling in neural machine translation, while GPT-style systems later used byte-level variants so the vocabulary could handle arbitrary strings more robustly.

In BPE, the training algorithm repeatedly merges the most frequent adjacent symbol pair. Start with characters, count pair frequencies, merge the most common pair into a new unit, then iterate. Over time, frequent morphemes and common word fragments become atomic tokens. The consequence is that “tokenization” is not whitespace splitting. A single visible word may become one token, several tokens, or part of a larger token depending on the tokenizer’s learned vocabulary.

Coder View

vocab = set(characters)
splits = {word: list(word) for word in corpus}

for step in range(num_merges):
    pair_counts = count_adjacent_pairs(splits)
    best_pair = argmax(pair_counts)
    vocab.add(join(best_pair))
    splits = merge_pair_in_all_words(splits, best_pair)

def tokenize(word, vocab):
    return greedily_match_longest_subwords(word, vocab)

Why this matters operationally: context windows are measured in tokens, not words; API billing is token-based; and detector systems that compute perplexity or log probabilities must use a tokenizer-model pair. Change the tokenizer and you can shift the statistics.

Plain English

The AI does not truly “see words” the way people do. It sees pieces. Some pieces are full words. Some are chunks like ing, un, punctuation, or even strange byte fragments. That is why one short sentence can be more tokens than a longer-looking sentence, and why pricing and context limits can feel weird.

If a model breaks text into chunks differently, it may also judge that text differently.

What We Can Only Infer

[INFERRED] Commercial detectors that mix classifier signals with perplexity-style signals probably standardize preprocessing carefully, because inconsistent normalization and tokenization would make sentence-level highlighting unstable. Vendors rarely disclose this in full detail, but any production detector needs it.

Chapter 4: Perplexity — The Famous Metric That Explains Less Than Marketing Suggests

Entropy, cross-entropy, surprisal, and why low perplexity is useful but not dispositive.
Math

Professor View

Perplexity measures how well a probability model predicts a sequence. In practice, it is the exponential of average negative log-likelihood, often written as PP = exp(-1/N Σ log P(xᵢ | x<i)). Lower perplexity means the model is less “surprised” by the text. Because LLM outputs are sampled from distributions they themselves define, they often land in relatively high-probability regions of model space, yielding lower perplexity under similar models. Human writing is often less distributionally compliant, especially when it contains idiosyncratic phrasing, novel emphasis, abrupt topic shifts, or mixed stylistic signals.

But perplexity is not a truth oracle. Simple human prose can also have low perplexity. Formulaic institutional writing can have very low perplexity. Heavily edited AI text can rise in perplexity. Text translated into simpler English can look more AI-like. This is why modern detectors either mix perplexity with additional features or move to classifier-based systems entirely.

Coder View

def perplexity(log_probs):
    # log_probs = token log P(token | prefix)
    avg_neg_log_likelihood = -sum(log_probs) / len(log_probs)
    return exp(avg_neg_log_likelihood)

example_log_probs = [-0.3, -0.7, -1.1, -0.2]
pp = perplexity(example_log_probs)

DetectGPT pushed beyond raw perplexity by checking whether a passage lies in a negative-curvature region of the model’s log-probability landscape. That is a zero-shot detection approach, not a commercial turnkey workflow by itself, but it clarifies why “predictability” is broader than a single scalar score.

Plain English

Perplexity asks: “How shocked is the model by this text?” If the text looks like the kind of thing a language model would naturally say, the model is not very shocked. If the text is more unusual, surprising, or oddly human, the model is more shocked. That sounds powerful, but lots of human writing is predictable too, especially school essays, templates, and polite workplace prose.

What We Can Only Infer

[INFERRED] When vendors still mention burstiness and perplexity in public explanations, that does not mean the production system is only using those. It often means those concepts are understandable marketing proxies for a more complex ensemble of classifiers, heuristics, and post-processing layers.

Chapter 5: Burstiness, Stylometry, and What Makes Text Feel Human

Variance, rhythm, vocabulary richness, mixed authorship, and why fairness is a serious technical problem.
Fairness

Professor View

Stylometry studies measurable features of writing style: type-token ratio, sentence-length variance, punctuation habits, part-of-speech frequencies, discourse markers, pronoun patterns, and more. Detector marketing often uses “burstiness” as a shorthand for sentence-length and complexity variation. Human text tends to vary more across clauses, tone, pacing, and rhetorical density. Raw AI text, especially under conservative decoding, often exhibits smoother local distributions and lower variance.

[INDEPENDENT] Yet fairness concerns are not incidental. Liang et al. (2023) showed that seven widely used detectors mislabeled non-native English TOEFL essays as AI-generated at an average false-positive rate of 61.3%, while performing far better on US eighth-grade essays. The problem is structural: simpler word choice and lower-perplexity syntax can be misread as “machine-like” even when the text is fully human. This turns style, proficiency, or neurotype into a risk factor.

Coder View

features = {
  "mean_sentence_len": mean(sentence_lengths),
  "sentence_len_std": std(sentence_lengths),
  "type_token_ratio": unique_tokens / total_tokens,
  "hapax_rate": count_once(tokens) / total_tokens,
  "punctuation_profile": histogram(punctuation_marks),
  "pos_distribution": tag_distribution(tokens),
  "discourse_markers": count_markers(tokens),
  "revision_trace": metadata_if_available
}
score = classifier.predict_proba(features)

Metadata changes the game. Revision histories, keystroke traces, and source attribution layers such as Grammarly Authorship provide process evidence, not just surface-text evidence. That is often more probative than stylometry alone.

Plain English

Humans are messy. We speed up, slow down, repeat ourselves, then suddenly write one sentence that sounds nothing like the last three. Detectors look for places where writing seems too smooth, too even, too statistically tidy. The trouble is that some real people also write in a simple, controlled, or formal way, so the detector can confuse “clear human writing” with “AI writing.”

What We Can Only Infer

[INFERRED] Detector vendors that add mixed-authorship categories, sentence heat maps, or authorship-replay features are tacitly acknowledging that binary human-vs-AI framing is no longer enough. The real-world problem is now composite writing: brainstormed by AI, drafted by AI, edited by a person, then partially rewritten by a humanizer or assistant.

Chapter 6: How AI Detectors Actually Work

Classifier stacks, zero-shot statistics, watermarking, highlighting, thresholds, and why no detector can be perfect.
Pipeline

Professor View

There are three broad families of detection methods. First, supervised classifiers trained on human and machine-written corpora, often transformer-based, which learn distributional cues directly from labeled data. Second, zero-shot or statistic-driven methods, such as perplexity or DetectGPT-style curvature analysis, which do not require a separate detector fine-tuning stage. Third, watermarking, which is proactive rather than forensic: it embeds a detectable signal during generation rather than inferring one later from generic style. Kirchenbauer et al. (2023) show how token “green lists” and a statistical test can create detectable provenance with modest quality impact under appropriate conditions.

Commercial systems usually operationalize detection as a pipeline: ingest text, normalize, extract multi-scale features, run one or more scoring models, map scores into UI categories, then generate highlights or confidence bands. The UI layer matters. A 70% AI score is not self-explanatory unless the vendor clearly states whether it means document-level probability, fraction of qualifying text, or a heuristic confidence band.

Coder View

input_text
  -> normalize()
  -> segment(document, paragraph, sentence)
  -> extract(
       embeddings,
       perplexity_stats,
       stylometry,
       source_overlap,
       revision_metadata_if_any
     )
  -> ensemble_score()
  -> threshold_map()
  -> highlight_sentences()
  -> render(report)

False positives and false negatives are inevitable because the classes overlap. Detector outputs therefore depend on threshold choices, class priors, and the operational cost of being wrong. A vendor that optimizes for recall will flag more true AI text but also more human text. A vendor that optimizes for low false positives may miss more AI-assisted writing.

Plain English

Most detectors do not “know” who wrote something. They measure patterns, compare those patterns to examples they have seen before, and make a guess. Then they turn that guess into a score, label, or highlighted paragraph. If two kinds of writing overlap too much, the detector can only make a cautious estimate, not a certainty.

What We Can Only Infer

[INFERRED] Vendors that claim both broad model coverage and resistance to bypassers likely run frequent model updates, maintain adversarial test sets, and use ensemble logic rather than one monolithic classifier. That is the only scalable way to keep up with new LLMs and paraphrasing tactics.

Chapter 7: How AI Humanizers Work

Counter-detection rewriting, semantic preservation, and why “humanized” is not the same as “human-authored.”
Adversarial

Professor View

AI humanizers can be thought of as adversarial style-transfer systems. Some operate as surface paraphrasers: they swap synonyms, alter sentence order, and vary transitions. Others present themselves as detector-aware, aiming specifically to perturb the statistical signals that detectors use, such as uniformity, repeated framing, low sentence-length variance, or highly regular rhetorical structure. Stronger systems likely mix semantic preservation objectives with anti-detection objectives.

The core tension is quality degradation. The more aggressively a humanizer pushes text away from a detector’s learned AI region, the greater the risk of introducing semantic drift, awkward wording, logical distortion, fake specificity, or citation mismatch. That is why many humanizer products simultaneously promise “same meaning” and “pass detectors”: they are selling a difficult optimization problem as a one-click UI event.

Coder View

def humanize(text, mode):
    plan = analyze(text,
                   tone=True,
                   repetition=True,
                   sentence_variance=True,
                   detector_risk=True)
    candidate = rewrite(text, style_mode=mode)
    candidate = enforce_constraints(candidate,
                                    preserve_meaning=True,
                                    vary_rhythm=True,
                                    reduce_ai_tells=True)
    maybe_rescore(candidate)
    return candidate

Some public products explicitly advertise built-in rescanning against external detectors. That suggests a loop of generate -> estimate detector risk -> rewrite again, whether via internal heuristics or external APIs.

Plain English

A humanizer is basically trying to take AI writing that sounds too smooth or too generic and make it sound less predictable. It may break up long patterns, swap out common phrases, vary the rhythm, or push the tone toward something more casual, academic, or personal. But even if it sounds more human, that does not prove a human wrote it.

What We Can Only Infer

[INFERRED] The strongest commercial humanizers probably win less by “understanding humanity” and more by learning what current detectors dislike. In other words, they are not solving authorship; they are solving classification boundaries.

Part IV: Synthesis, Practical Guidance, and Forecast

This is the “paper” layer of the report: the argument, the caution, and the forward-looking interpretation.

Research-Paper Thesis

[CITED] AI detectors produce probabilistic estimates of distributional similarity, not proof of authorship. That distinction is not semantic hair-splitting. It is the difference between “this text shares patterns with machine-generated text under our current model” and “a machine wrote this.” The former is a defensible statistical statement. The latter is a social accusation that text alone often cannot justify.

The impossibility argument is practical as much as theoretical. Human writing and AI writing increasingly overlap; AI outputs can be heavily edited; human writing can be formulaic; detectors are trained on evolving models; and adversarial rewrites continuously move the boundary. Perfect text-only authorship inference would require a class separation that the real world does not consistently provide.

The ESL Bias Problem

[INDEPENDENT] Liang et al. found an average 61.3% false-positive rate on human TOEFL essays in a seven-detector study, compared with high accuracy on US eighth-grade essays. This is one of the most important findings in the literature because it shows that detector error is not random noise. It can be structured by language proficiency and writing style.

Mixed Authorship Is Now Normal

The dominant real-world class is no longer “pure human” vs “pure AI.” It is collaborative text: brainstormed with AI, drafted by AI, rewritten by human, edited by grammar tools, cited by assistance tools, and maybe humanized to avoid false flags. Products that still pretend the world is binary are already behind reality.

Why Humanizers Exist

They exist partly because detectors overclaim. If a detector’s score is treated as a verdict, users will seek tools that perturb the score. Some of that use is deceptive. Some of it is defensive, especially when users fear false accusations. The market grew in the space between technical uncertainty and institutional overconfidence.

Practical Guide

For Students

A score is not guilt. If you are flagged, preserve drafts, revision history, notes, prompts, and timestamps. Process evidence is often stronger than arguing over a single detector’s number.

For Educators

Use detector output as one signal among many. Ask for drafts, oral explanation, source notes, or Authorship-style evidence. Do not convert a probability into a disciplinary certainty.

For Content Teams

Decide what you are actually policing: undisclosed AI use, poor quality, factual risk, or plagiarism. Different tools optimize for different governance problems.

For Technical Evaluators

Benchmark on your own text classes. Long-form academic prose, marketing copy, translated text, and collaborative documents behave differently. Vendor benchmark claims are not deployment truth.

For Policy Makers

Support provenance and disclosure systems, but avoid policies that treat current detectors as forensic certainty. The technology is still too probabilistic and too biased for that.

Decision Rule

If the consequence is serious, text-only detection should never be the only evidence. The higher the stakes, the more process evidence you need.

Forecast

  • Watermarking and provenance will matter more, but only if major generation platforms adopt them consistently and users do not rewrite away the signals.
  • Detector vendors will increasingly train on paraphrased and humanized AI rather than raw ChatGPT-style output.
  • Authorship and replay systems will become more valuable because they shift evidence from style to process.
  • The market will keep converging: writing assistants will add transparency, detectors will add rewrite suggestions, and humanizers will add built-in scanning.
  • The arms race has no stable endpoint because each side changes the statistical landscape the other side learns from.

Glossary

TokenA unit in a model’s vocabulary. It may be a whole word, part of a word, punctuation, or a byte-level fragment.
PerplexityA measure of how surprised a model is by a text sequence. Lower means the text is easier for the model to predict.
BurstinessA rough public shorthand for variation in sentence length, complexity, and rhythm across a passage.
StylometryQuantitative analysis of writing style using lexical, syntactic, and discourse-level features.
WatermarkingEmbedding a detectable signal during text generation so later systems can test for that signal.
Mixed AuthorshipText created through a combination of human writing and AI assistance rather than one pure origin.
ClassifierA model that assigns text to categories such as likely human, likely AI, or mixed.
Top-p / Nucleus SamplingA generation method that samples from the smallest token set whose cumulative probability exceeds a chosen threshold.
